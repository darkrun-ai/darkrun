CORRECTNESS review of the build station's 4-unit decomposition (fixture-schema, sim-spine, replay-route, ci-gates) ‚Äî three confirmed defects, all in machine-executed `quality_gates` commands that will reject a spec-compliant implementation or silently pass on a broken one. None of these are pre-existing/locked-spec issues (verified: spec.md never asserts any of the assumptions these gates hardcode); all three are introduced by the unit decomposition itself.

## 1. [sim-spine, HIGH] `fixture-committed` gate asserts a JSON casing that a spec-compliant serde impl will never produce

`sim-spine.md` frontmatter:
```
- name: fixture-committed
  command: sh -c 'test -s crates/darkrun-sim/fixtures/dark-core.json && python3 -c "import json;d=json.load(open(\"crates/darkrun-sim/fixtures/dark-core.json\"));assert d[\"schema_version\"]==1 and d[\"outcome\"]==\"sealed\" and len(d[\"ticks\"])>0 and len(d[\"units\"])>0"'
```

This hardcodes `d["outcome"]=="sealed"` (lowercase). But `fixture-schema.md` (the sibling unit that owns the `FixtureOutcome` type) explicitly instructs: *"`FixtureOutcome` uses serde's externally-tagged default unless Contract 3's text specifies a tagging ‚Äî match whatever the spec's Rust block shows."* Contract 3's Rust block (`spec.md` lines 216-219) shows the enum with NO `#[serde(rename_all=...)]` attribute:
```
pub enum FixtureOutcome {
    Sealed,
    Escalated { reason: String },
}
```
I compiled this exact enum with `serde_derive` (default tagging, no rename) and serialized it:
```
Sealed -> "Sealed"
Escalated -> {"Escalated":{"reason":"x"}}
```
So a `Sealed` unit variant serializes to the JSON string `"Sealed"`, not `"sealed"`. Every OTHER enum in this codebase that derives Serialize (`Status`, `RunAction`, `CheckpointKind`, etc. in `domain.rs`/`position.rs`) carries an explicit `#[serde(rename_all = "snake_case")]` ‚Äî but Contract 3's block for `FixtureOutcome` conspicuously omits it, and `fixture-schema.md` tells the implementer to match the block exactly. I grepped `spec.md` for `"sealed"`/`rename_all`/`d["outcome"]` ‚Äî none appear anywhere; this lowercase assumption originates in `sim-spine.md`'s own gate, not the locked spec.

Effect: a correct, spec-compliant `fixture-schema` + `sim-spine` implementation produces `{"outcome":"Sealed",...}`, and the `fixture-committed` gate's `assert d["outcome"]=="sealed"` raises `AssertionError` and fails ‚Äî rejecting correct work. (`sim-spine.md`'s own prose completion-criterion #6 hedges correctly ‚Äî *"outcome == \"sealed\" (or the exact serde tag Contract 3's enum produces ‚Äî match the schema)"* ‚Äî but the machine-executed frontmatter gate does not hedge; it hardcodes the wrong literal.)

Fix: either add the missing tagging decision to Contract 3 (out of this review's authority to invent) is not needed ‚Äî simplest fix within unit-spec authority: make the `fixture-committed` command assert on the deserialized Rust value instead of raw JSON string comparison (e.g. drive it through a tiny helper or just check `"outcome" in d` and rely on the crate-tests/committed_fixture_matches_regeneration test for the real assertion), or fix the literal to `"Sealed"` to match what `fixture-schema.md`'s own instruction will actually produce.

## 2. [sim-spine, HIGH] `seam-greps` gate's zero-tolerance `.action` ban is stricter than AC-3 and conflicts with Contract 1's own dictated text

`sim-spine.md` frontmatter:
```
- name: seam-greps
  command: sh -c '! grep -n "[^_]run_tick(" ... && ! grep -n "\.action" crates/darkrun-sim/src/world.rs crates/darkrun-sim/src/provider.rs && ...'
```

This demands ZERO occurrences of the substring `.action` anywhere in `world.rs`/`provider.rs` ‚Äî comments and doc-comments included, since grep is purely textual. But spec.md's own AC-3 check (1) is explicitly NOT a zero-tolerance check: *"grep -n '\.action' <world module> <provider module> shows every match either absent from the decision path, or confined to a function whose name signals post-hoc grading (e.g. containing grade, project, or transcript) ‚Äî never inside the function that calls Provider::next_move..."* AC-3's positive-boundary text (spec.md line 27-29) and Contract 1 (spec.md line 176-178) both explicitly PERMIT `.action` reads in *"the world module's own grading step that runs after a ProviderMove has already been chosen and executed"* ‚Äî which is exactly what `world.rs` needs to do per this SAME unit's own Edge-case requirements: AC-9/`escalate_scenario_is_detected_red` requires the world to detect `RunAction::Escalate` on the returned `TickResult` to report `FixtureOutcome::Escalated`, and the "Post-`Sealed` ticks" edge case requires *"The world module's tick loop stops issuing run_tick_with_hosting calls the FIRST time it observes Sealed."* Both require reading `tick.action` (or an equivalent field access containing the literal substring `.action`) in `world.rs` itself ‚Äî and the zero-tolerance grep will flag it.

Compounding this: Contract 1's Rust block, which `sim-spine.md` instructs to copy "verbatim" into `provider.rs`, contains a doc comment that literally reads *"The signature carries no parameter for `TickResult.action`"* ‚Äî this text alone (if copied as instructed) trips the `.action` grep in a file the gate scans.

`sim-spine.md`'s own module-by-module section 2 asserts, incorrectly stronger than the spec: *"`.action` never appears in world.rs/provider.rs ‚Äî the seam-greps gate enforces the textual half"* ‚Äî this overstates AC-3's actual (narrower) requirement and bakes the overstatement into the automated gate. The only way to satisfy this literal gate is to avoid ever writing `<expr>.action` field-access syntax (e.g. destructure `TickResult` on return and match a locally-named variable instead) and to avoid quoting Contract 1's own doc comment verbatim ‚Äî a non-obvious workaround the unit spec neither hints at nor sanctions, and one a natural implementation (matching the existing `harness.rs` style of `tick.action`, `decided.action` field access this very unit is modeled on) will not stumble into by luck.

Fix: scope the seam-grep the same way AC-3's own check does ‚Äî e.g. exclude matches inside functions whose names contain `grade`/`project`, or drop this half of the automated gate and rely on manual/reviewer verification of AC-3's structural rule (as spec.md itself does for check (2), calling it "a named review assertion, not grep-automatable").

## 3. [ci-gates, MEDIUM] `local-dry-run` gate vacuously passes if the named tests are missing or renamed

`ci-gates.md` frontmatter:
```
- name: local-dry-run
  command: sh -c 'cargo test -p darkrun-sim committed_fixture_matches_regeneration && cargo test -p darkrun-sim escalate_scenario_is_detected_red'
```

`cargo test -p <pkg> <name-filter>` exits 0 with "0 tests run" when nothing matches the filter ‚Äî it does not error. I confirmed this empirically against the current tree (pre-work, before either test exists):
```
$ cargo test -p darkrun-sim committed_fixture_matches_regeneration
running 0 tests
test result: ok. 0 passed; 0 failed; ...
$ echo $?
0
```
So if `sim-spine`'s implementation ships a differently-named test, an `#[ignore]`d test, or simply omits one of these two named tests (a typo or a dropped requirement), `ci-gates`'s `local-dry-run` gate ‚Äî and by extension the sim-fidelity CI job it wires up ‚Äî will report success anyway. The sibling `job-steps-present` gate only checks that the LITERAL STRING appears somewhere in `ci.yml`'s step text (which `ci-gates` itself writes, trivially true), not that the referenced test exists and runs. Nothing in `ci-gates.md`'s own automated gates proves AC-16's core claim ‚Äî *"AC-9's dedicated escalate-scenario test proves the same detection path the CI job depends on actually fires"* ‚Äî actually holds; it only proves it holds IF the test happens to exist under that exact name, which this unit's own gates don't verify.

This is meaningfully mitigated (not eliminated) by `depends_on: [sim-spine, replay-route]` ‚Äî by the time `ci-gates` builds, sim-spine's own (manually-verified, prose) completion criterion #2 requires the five named tests to exist and run non-trivially. But that upstream check is human-verified prose, not machine-gated, so a slip there (test renamed in a later fix-feedback pass, e.g.) would sail through `ci-gates`'s own automated verification undetected.

Fix: replace the bare filtered `cargo test` calls with a check that also asserts a nonzero test count, e.g. `cargo test -p darkrun-sim committed_fixture_matches_regeneration 2>&1 | grep -qE '[1-9][0-9]* passed'` (and the same for the escalate test), so a missing/renamed test fails loudly instead of silently reporting green.

All three findings are self-contained corrections to the unit specs' own machine-checked gates; none require reopening the frame's or spec's locked decisions (the 4-unit chain, the fb-08 units-field amendment, the dedicated sim-fidelity CI job, crate-partition/fixture-schema-placement/determinism-by-normalization are untouched by these fixes).
