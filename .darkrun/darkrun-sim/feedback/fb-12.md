## The defect: sim-spine `seam-greps` final clause always passes once world.rs exists

`sim-spine.md` frontmatter, final clause of `seam-greps`:
```
test "$(grep -c \"\\.action\" crates/darkrun-sim/src/world.rs)" = "$(sed -n \"/fn grade_tick/,/^}/p\" crates/darkrun-sim/src/world.rs | grep -c \"\\.action\")"
```

The whole gate is wrapped in `sh -c '...'` (single quotes), so the `\"` sequences reach the inner sh literally. Inside a `$(...)` command substitution, parsing starts a FRESH quoting context — the enclosing double quotes do not apply — so `\"` becomes a literal `"` CHARACTER in the argument, not a quote delimiter. Empirically verified (macOS sh):

- LHS: grep's pattern argument becomes `"\.action"` — WITH literal double-quote characters. Rust source (`tick.action`) contains no quotes around `.action`, so `grep -c` returns 0 for any real world.rs. Probe output: `printf "[%s] " \"\\.action\"` inside the same context prints `["\.action"]`.
- RHS: sed's script argument becomes `"/fn grade_tick/,/^}/p"` — leading/trailing literal quote chars make it an invalid sed script. sed errors (`sed: 1: ""/fn ...": invalid command code "`), emits nothing, and `grep -c` on empty stdin prints 0.
- `test "0" = "0"` → true. The clause passes regardless of world.rs content.

Demonstration with two mock world.rs files, running the exact clause verbatim:
- CASE A (compliant: all `.action` inside a top-level `fn grade_tick` ending with a `^}` line): exit 0.
- CASE B (VIOLATING: `if tick.action.is_spec() { self.seed(); }` inside the tick loop, outside grade_tick): exit 0 — the gate PASSES the violation.

So the machine gate protecting AC-3's core seam (the entire point of the sim: decisions never keyed on `.action`) enforces nothing for world.rs; only prose completion-criterion 4 (human-verified) covers it. This is the same vacuous-pass class as fb-10 item 3, reintroduced by the fix for fb-10 item 2. (Note the sibling `fixture-committed` gate's `\"` usage is NOT affected — there the `\"` sit inside plain double quotes at the sh level, where `\"` correctly yields a quote character for python; the bug is specific to `\"` inside `$(...)`.)

## Verified working replacement (drop-in, tested against both mock cases)

Avoid backslash-quote inside `$()` entirely — plain double quotes parse correctly in the fresh `$()` context even within the single-quoted `sh -c` string, and `[.]` sidesteps needing an escaped dot:
```
test "$(grep -c "[.]action" crates/darkrun-sim/src/world.rs)" = "$(sed -n "/^fn grade_tick/,/^}/p" crates/darkrun-sim/src/world.rs | grep -c "[.]action")"
```
Tested verbatim inside `sh -c '...'`: CASE B (violating) exits 1, CASE A (compliant) exits 0, and with world.rs absent it exits 1 (LHS empty vs RHS "0"), so it also fails correctly mid-work.

Two deliberate hardenings in the replacement, adopt both:
1. `[.]action` instead of `\.action` — same match semantics (literal dot + action), no backslash to mangle.
2. `/^fn grade_tick/` (anchored) instead of `/fn grade_tick/` — the unanchored form starts the sed range at the FIRST line containing the text `fn grade_tick`, which a module doc comment mentioning the function (likely, since the unit body tells the implementer about it) would trigger, truncating the extraction at the first `^}` before the real function and failing a compliant implementation. The unit body already pins grade_tick as "a single top-level function", so `^fn grade_tick` is exact. Mirror the same anchored form in completion-criterion 4's prose if it names the extraction.
