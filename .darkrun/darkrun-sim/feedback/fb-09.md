Maintainability review (build decomposition, role: maintainability) — FAIL on one consolidated cross-unit coherence defect.

**Finding: `FixtureOutcome`'s serde tagging is left ambiguous by fixture-schema, but sim-spine's own machine-checked quality gate hardcodes an assumption that ambiguity contradicts by default — verified empirically.**

`fixture-schema.md` ("What to build" item 3) instructs: "`FixtureOutcome` uses serde's externally-tagged default unless Contract 3's text specifies a tagging — match whatever the spec's Rust block shows." Contract 3's Rust block for `FixtureOutcome` (`spec.md` lines 216-219) shows no `#[serde(...)]` attribute at all. Read literally, this tells the Worker to add no `rename_all`/rename attribute to `FixtureOutcome`.

I verified empirically (`cargo run` against a throwaway crate with the exact `enum FixtureOutcome { Sealed, Escalated { reason: String } }` shape and a bare `#[derive(Serialize)]`, no other attributes) that serde's default externally-tagged representation serializes the fieldless `Sealed` variant as the JSON string `"Sealed"` (Rust-cased), not `"sealed"`.

`sim-spine.md`'s own frontmatter `quality_gates` entry `fixture-committed` runs:
```
python3 -c "import json;d=json.load(open(\"crates/darkrun-sim/fixtures/dark-core.json\"));assert d[\"schema_version\"]==1 and d[\"outcome\"]==\"sealed\" and len(d[\"ticks\"])>0 and len(d[\"units\"])>0"
```
— a hard, non-negotiable assertion that the JSON value is the lowercase string `"sealed"`. Sim-spine's own completion-criterion #6 hedges in prose ("`outcome == \"sealed\"` (or the exact serde tag Contract 3's enum produces — match the schema)"), acknowledging the ambiguity — but the *executable* gate that actually blocks the unit does not hedge: it will fail on a `"Sealed"` value.

Note also: `spec.md` itself never once writes the literal JSON string `sealed` (lowercase) anywhere — every reference to the outcome is the Rust-cased `Sealed`/`Escalated`/`FixtureOutcome::Escalated` — so the locked spec text gives no signal toward lowercase either.

Contrast with the rest of `darkrun-core/src/domain.rs`: every other Serialize/Deserialize enum in the crate (`Status`, `StationPhase`, `Surface`, etc.) explicitly carries `#[serde(rename_all = "snake_case")]`. That is the crate's established idiom for exactly this situation. Fixture-schema's instruction to add nothing beyond what Contract 3's bare code block shows works against that established idiom instead of invoking it, and nothing in either unit spec tells the fixture-schema Worker to reach for `rename_all = "snake_case"` on this one new enum.

**Concrete failure scenario:** The fixture-schema Worker builds `sim_fixture.rs` exactly to spec — no extra derive attribute on `FixtureOutcome`, since fixture-schema's own text discourages adding anything Contract 3's block doesn't show. Fixture-schema's own tests (round-trip equality) pass regardless of casing, so nothing catches the gap at that unit. The sim-spine Worker then builds the dark-mode fixture per that schema; `outcome` serializes to `"Sealed"`. Sim-spine's own `fixture-committed` quality gate fails on an otherwise fully correct, spec-compliant fixture. Whoever debugs it has no single source of truth in either unit spec telling them whether the fix belongs in fixture-schema (add `#[serde(rename_all = "snake_case")]`, matching domain.rs convention) or in the gate script — an avoidable rework loop caused by the decomposition, not by either Worker's error.

**Suggested resolution** (not mine to mandate, flagging for the operator/spec-author): fixture-schema should explicitly require `#[serde(rename_all = "snake_case")]` on `FixtureOutcome`, matching the domain.rs convention used by every other tagged enum in the crate, and sim-spine's `fixture-committed` gate / completion-criterion #6 should then drop the hedge now that the casing is pinned. Either resolution is fine — what isn't fine is leaving both units to independently guess and disagree.
