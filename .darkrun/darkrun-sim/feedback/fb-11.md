**New defect: sim-spine's reworked `seam-greps` gate — the grade_tick confinement leg is vacuous. It can NEVER fail, verified empirically.**

The gate (sim-spine.md frontmatter, `seam-greps`) ends with:
```
test "$(grep -c \"\\.action\" crates/darkrun-sim/src/world.rs)" = "$(sed -n \"/fn grade_tick/,/^}/p\" crates/darkrun-sim/src/world.rs | grep -c \"\\.action\")"
```
The `\"` escapes are the bug. The whole script sits inside the single-quoted `sh -c '...'` argument, where backslashes pass through literally; inside `$( )` the shell then parses `\"\\.action\"` in a fresh quoting context, so grep's pattern becomes the literal string `"\.action"` — WITH embedded double-quote characters. I reproduced this exactly: `grep -c \"\\.action\"` inside `$()` returns 0 against a file containing two real `.action` reads (the same file returns 2 with correct quoting). Both sides of the `test` therefore evaluate to "0" = "0" and the check passes unconditionally. The sed expression breaks the same way — `\"/fn grade_tick/...\"` word-splits at the space into two args and sed errors (`invalid command code`) to stderr, which command substitution swallows.

Proof: I built a synthetic world.rs with a blatant violation (`match tick.action` inside a decision-path function `decide_next`, outside `grade_tick`) and ran the gate's exact command verbatim: **exit 0 — green on the violation**. The correctly-quoted variant (plain `"..."` instead of `\"...\"` inside the `$( )`) exits 1 on the violation and 0 on a clean file where all `.action` reads sit inside `grade_tick` — the intended semantics work fine once the quoting is fixed.

This is the same defect class as fb-09 — executable gate disagreeing with the unit's prose (criterion 4: "every `.action` occurrence in world.rs falls inside the single `fn grade_tick` body") — but worse in direction: fb-09's gate was spuriously red (caught at build time); this one is silently green forever, giving false assurance on exactly the AC-3 seam the fb-10 amendment set out to machine-enforce. A future edit that leaks an `.action` read into world.rs's decision path sails through.

**Fix:** replace the `\"` escapes inside the `$( )` substitutions with plain double quotes — inside the single-quoted sh -c string no escaping is needed:
```
test "$(grep -c "\.action" crates/darkrun-sim/src/world.rs)" = "$(sed -n "/fn grade_tick/,/^}/p" crates/darkrun-sim/src/world.rs | grep -c "\.action")"
```
(verified working in both directions, violation and clean).

Two secondary notes to fold into the same edit, both non-blocking: (1) the sed range opens at the FIRST line matching `fn grade_tick` and re-opens at any later match — a test named e.g. `fn grade_tick_stops_on_terminal()` would widen the counted region; anchoring the pattern (e.g. `/^fn grade_tick(/` or `/fn grade_tick(/`) cheaply closes that. (2) Module 2 demands the confinement rule in spec prose and the gate but not as a doc comment on `grade_tick` itself; since the gate stops enforcing after this unit completes (CI only runs the two named tests + clippy), a one-line demand that `grade_tick`'s doc comment state "all .action reads in world.rs must live in this function; the seam gate counts them" would give the future maintainer the constraint at the code site. The function name `grade_tick` itself is convention-clean (verb_noun snake_case, satisfies AC-3's "name signals post-hoc grading" rule) — no issue there.

Everything else in the round's edits re-checked clean. This gate line is the one blocking issue.
