Gap in author-spec.md's Required document structure: Phase 2 (replay player) is under-specified relative to what the frame locks in and what the unit's own "Verified facts" section already found.

Frame's Build order Phase 2 (frame.md lines 60-61) locks in three concrete commitments beyond "a route exists": (a) the new Route is modeled on `web/site/src/pages/preview.rs`'s pattern specifically (static/hardcoded payload, no fetch, explicit no-live-feed banner) rather than `browse.rs`'s live-fetch pattern; (b) panels are composed from the shared `darkrun-ui` component set so the replay view matches the site's existing visual language; (c) the route is added to `route.rs`'s `all_paths()` static-render list (implied by "Demoable independently: load the replay route... with no engine process running" — a wasm static export only reaches routes registered there).

The unit's own "Verified facts to specify against" section (line 43) names these exactly: "new Route variant in `web/site/src/route.rs` (and its `all_paths()` static-render list), modeled on `/preview`... composed from darkrun-ui prelude components (`StationStrip`, `StationPipeline`... `UnitGraph`...)." But unlike the CI-gate-shape fact in that same sentence — which gets an explicit "the spec must state the Phase 3 CI gate shape" imperative — nothing here says "the spec must state" the Route's contract. And when you check where the imperative demands actually land:

- Required document structure §2 (Acceptance criteria, "cover at minimum" list) has exactly one Phase-2 bullet: "replay route renders the committed fixture." No mention of darkrun-ui component reuse or all_paths() registration.
- Required document structure §3 (Contracts) is a closed, non-"at minimum" list of five items: provider trait, NoopHosting, fixture schema, crate module map, fixture embedding path/mechanism. No Route/UI-component contract line item at all.

Net effect: a builder implementing Phase 2 strictly off spec.md has no demanded contract or AC forcing (1) reuse of the specific darkrun-ui components (StationStrip/StationPipeline/UnitGraph) instead of hand-rolled markup, or (2) registering the new route in `all_paths()`. Locked decision 4 (restated verbatim per the unit's own instruction) blocks the worst case — a live-fetching route — but does not cover component reuse or static-render registration. Omitting all_paths() registration in particular would let a route "exist" and even render in dev mode while silently never reaching the static wasm export, which is exactly what Phase 3's CI gate is supposed to validate ("the site build consuming the committed copy... green"). That's a hole a builder would have to fill by guessing.

Fix: add a Contracts-section requirement (or extend the existing closed list) demanding spec.md pin the Route's shape — modeled on /preview (static payload, no fetch, no-live-feed banner), composed from the named darkrun-ui components, and registered in `all_paths()` — with a corresponding AC whose check command verifies the route appears in the static-render list (not just that it compiles).
